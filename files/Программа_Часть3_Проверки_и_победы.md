## М3. Написание отчёта: замечания и стиль (55 мин)

### Подцель
Агент-редактор оформляет замечание по стандарту фирмы. Проверяемость важнее гладкости.

### LMS-опора
- **Структура замечания (3 блока):** Описание ситуации → Требования НПА → Выводы и
  рекомендации.
- **Железные правила Памятки:** счета полным наименованием (не «Д-т 08 Кр-т 07») ·
  НПА дословно, актуально на период, точный пункт · не толковать норму от себя ·
  только сам акт, без комментариев (Илекс, Бизнес-инфо) · первая ссылка = полное
  наименование + кем утверждён + дата + № + «(далее — …)» · первичка: дата, номер,
  для ТТН/ТН — серия · выводы: конкретные исправления, влияние на налоги и отчётность,
  вывод о существенности.
- **Шпаргалка «Нормы языка»** (числа 18 359,75; даты без «г.»; «ёлочки»; «Беларусь»).
- **Промпт редактора замечаний** (`Промпты_агентов_Gem`, №3); файлы-знания: Памятка,
  Нормы языка, образец «стандарт».
- **Два образца** замечаний (эталон и антипример).

### Демо (последовательность шагов)
1. Создать Gem «Редактор замечаний», приложить файлы-знания (Памятка, Нормы языка).
2. Подать черновик по счёту 97 «Расходы будущих периодов».
3. Разобрать вывод в 3 блока → сверить с эталоном «стандарт».
4. **Главный момент:** без текста нормы агент НЕ выдумывает закон — ставит плашку
   «[вставьте точную цитату НПА, актуальную на период аудита]».
5. Показать связку с Частью 2: цитату норм берём из NotebookLM и вставляем в блок 2.

### Практика (шаги)
1. Каждый берёт черновик ошибки → прогоняет через агент.
2. Доводит: вставляет норму (из правовой базы / NotebookLM), проверяет суждение.
3. **Чиним типовое:** агент пересказал норму вместо цитаты → напомнить «только
   дословно», проверить, что стоит плашка, а не выдуманный закон.

### Важно (проговорить)
Оформление (шрифты, цвет, поля) в агента НЕ зашиваем — это вёрстка Word, отдельный
стиль/мини-макрос. Агент отвечает за текст и структуру.

## М4. JET: проверка проводок + НПА (35 мин)

### Подцель
Механику проверки проводок делает макрос, суждение и выводы — агент. NotebookLM
(уже освоен в Части 2) — для работы с НПА.

### LMS-опора
- **Карта 13 тестов** с пометкой «макрос / агент»: тесты 1,3,4,5,6,7,9,10,11,12,13 —
  МАКРОС (арифметика); тесты 2 и 8 + написание выводов — АГЕНТ (суждение/текст).
- **JET-макрос** (`Макросы_весь_код`, №2) + учебный файл с подсеянными аномалиями.
- **Промпт «Вывод по JET-тесту»** (`Промпты_агентов_Gem`, №4).

### Демо (последовательность шагов)
1. Открыть учебный файл → импортировать/вставить JET-макрос → `ПрогнатьВсеТесты`.
2. Показать лист «Флаги»: как макрос «схлопнул» многострочную выгрузку 1С и разложил
   срабатывания. Отсортировать/отфильтровать по тесту.
3. Разобрать 2–3 флага. **Про Бенфорд:** на чистых данных распределение совпало с
   ожидаемым → тест, вернувший «норму», тоже результат, а не провал.
4. Взять флаг теста 8 (в файле подсеяно «возврат бракованного товара, исправление
   ошибки») → передать агенту «Вывод по JET-тесту» → показать формулировку вывода.
5. **Связка целого дня:** механика (макрос) → суждение (агент) → проверка (человек).
   Существенность — формула, не ИИ.

### Практика (шаги)
1. Каждый прогоняет макрос на своей копии учебного файла → находит флаг.
2. Передаёт флаг агенту → получает черновик вывода → правит под факт.

## М5. Быстрые победы (20 мин)

### Подцель
Два готовых Gem на каждый день + честная граница ИИ на примере плана.

### LMS-опора
- **Промпт «Деловое письмо»** (`Промпты_агентов_Gem`, №5) — 5 типов писем.
- **Промпт «Черновик структуры плана аудита»** (№6) — буфер.

### Демо + практика (последовательность шагов)
1. **Письмо:** один бриф → агент собирает письмо → меняем ТИП (1–5) → показать, как
   перестраивается каркас. Каждый делает свой Gem под свою частую задачу.
2. **План (если есть время):** заметки → агент даёт черновик-скелет участков.
   **Главный момент:** агент НЕ подставляет числа существенности — ставит плашку
   «[M/PM/TA — формула из ОСВ, не ИИ]». Это и есть капстоун дня: где ИИ помогает
   (структура) и где нельзя (числа существенности, коды РД).
   *Усиление:* приложи справочник кодов РД фирмы как файл-знание Gem — тогда агент
   подставит реальные коды вместо плашек (методология можно, данные клиента — нельзя).
